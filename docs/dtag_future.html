doodle tag.dtag notes

This document records the current DoodleTag behavior for sharing .dtag macro files and outlines the next larger sharing direction that may be pursued later.

Current Implementation

DoodleTag currently shares macros as raw .dtag files:

  • a user records or exports a macro
  • DoodleTag shares the raw .dtag file
  • another device with DoodleTag can open that file

This works best when:

  • both devices already have DoodleTag installed
  • the host app or system surface hands the file to DoodleTag correctly
  • the recipient is intentionally exchanging the macro file, not discovering it publicly

Current iOS Behavior

On iOS, one important distinction remains:

  • the Files app is good at handing a registered document type to its owning app
  • Messages and similar hosts are less predictable, because they often treat the file as an attachment inside their own UI instead of directly invoking the receiving app

To improve that intake path, DoodleTag now ships an iOS Share Extension:

  • DoodleTag can appear in the iOS share sheet for compatible .dtag file, file URL, URL, or generic data shares
  • the extension writes the accepted item into the shared app-group inbox
  • the main app imports the pending item when DoodleTag becomes active

This improves:

  • sharing from Messages-like hosts when the host exposes a share sheet
  • cases where the host app does not directly open the owning document app, but does allow sharing the attachment outward

It does not guarantee:

  • tapping an attachment bubble in Messages and jumping straight into DoodleTag

That last behavior depends on Apple host-app behavior and document handoff rules. The Share Extension improves the share-sheet route, not every attachment-preview route.

Current Product Position

  • macro sharing remains file-based
  • the primary shared artifact is the raw .dtag file
  • the iOS Share Extension is part of that installed-device file-sharing workflow
  • sharing is still one-to-one between devices that already use DoodleTag

Future Direction: Upload to One Server and Share the Link

The next larger direction would be:

  • upload the .dtag macro file to one public server
  • generate one openly accessible URL for that file
  • share the link instead of sharing the file itself

Proposed Flow

  1. A user records or exports a macro.
  2. DoodleTag uploads the .dtag file to one server.
  3. The server returns a public URL for that macro file.
  4. DoodleTag shares a normal web link that points to the uploaded macro.

That shared link could be:

  • the direct public .dtag URL
  • or better, a doodletag.500nits.com landing URL that references the uploaded .dtag

The landing-link version is stronger because it lets DoodleTag control the recipient experience.

Recipient Behavior in the Future Model

If the Recipient Has DoodleTag Installed

  • the shared link can route into DoodleTag through the existing iOS/URL handling path
  • DoodleTag can resolve the real .dtag
  • DoodleTag can download the file and show the normal import or play flow

If the Recipient Does Not Have DoodleTag Installed

  • the shared link opens in the browser
  • the browser can land on a DoodleTag page
  • that page can explain what the macro file is
  • that page can send the user to the App Store

This is the main growth effect of the link-based model:

  • the public share stops being useful only to existing DoodleTag users
  • that separate public flow can also expose the app to new users who do not yet have it installed

Why This Future Direction Is Attractive

This direction has several product advantages.

  • the shared object becomes easier to send through Messages, mail, social apps, and any other host that handles links well
  • the web page can explain what DoodleTag is and what the recipient should do next
  • the App Store page becomes part of the natural fallback path for people who receive the link without the app

The Server Side of That Future Direction

The simplest implementation shape is:

  • one public object-storage bucket
  • one random object path per uploaded macro
  • one public URL returned after upload

Examples of simple storage choices:

  • Cloudflare R2
  • Amazon S3
  • Supabase Storage

If ease of implementation is the main concern:

  • Supabase Storage is likely the easiest full developer experience
  • Cloudflare R2 is attractive for simple public object hosting and cost
  • S3 is fine if AWS is already part of the stack

A minimal object shape could be:

  • macros/2026/03/<random-token>.dtag

The important detail is that the file key should be long and unguessable.

Privacy and Product Implications

Public link sharing means:

  • anyone with the link can access the macro
  • recipients can forward the link to other people
  • the link may appear in previews, logs, analytics, or other third-party systems
  • if object keys are weak, the uploaded files may become guessable or enumerable

There is also macro metadata exposure to consider.

The current private .dtag format can contain information such as:

  • creator/device identity fields
  • creator display name
  • timestamps
  • stroke, tool, and event history

Minimum guardrails should be:

  • random, non-guessable object names
  • explicit UI wording that the generated link is public-by-link
  • a dedicated public-sharing export that strips creator identity, device-related fields, and any other sensitive or protected metadata before upload
  • later consideration for expiration or deletion

In other words:

  • publicly shared .dtag files should not be byte-for-byte copies of the private one-to-one exchange format
  • they should be sanitized before upload so the public share does not expose sensitive creator or device information

Recommended Future Shape

If this work is resumed later, the strongest product shape is:

  • keep direct .dtag file sharing for one-to-one installed-device exchange
  • add a separate upload-and-share-link flow for public sharing

That gives DoodleTag two distinct modes:

  • Share File
    • direct device-to-device exchange
    • best for installed users
  • Share Publicly
    • upload to one server
    • share a public URL
    • best for Messages, social sharing, and recipients who may not have DoodleTag installed
